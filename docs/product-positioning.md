# MaxC CLI 产品定位（基础 API 视角）

## 1. 核心判断

`maxc cli` 当前最有价值的定位，不是“另一个内建 agent”，而是：

- 面向 MaxCompute 的结构化执行层
- 面向外部 LLM / Bot / CI 的数据操作底座
- 面向人和机器同时可读的只读分析入口

换句话说：

- 推理由外部 agent 负责
- 执行、观测、约束、审计由 `maxc cli` 负责

这个定位比“先在 CLI 内堆一层内建编排器”更稳，因为它更贴近真实 MaxCompute 的基础能力，也更容易形成高频使用场景。

## 2. 为什么这个定位成立

只看当前已经具备或容易补齐的基础能力，`maxc cli` 已经能覆盖一个完整的分析闭环：

1. `meta search / describe / partitions`
   用来发现表、理解 schema、确认分区。
2. `data sample / profile`
   用来理解字段内容、空值情况和样例分布。
3. `query`
   用来执行只读 SQL，返回结构化 JSON 结果。
4. `job submit / status / wait / result`
   用来处理长查询和异步执行。

这意味着即便 CLI 不提供内建 agent 执行能力，一个外部 LLM 也已经可以基于 `maxc cli` 完成“理解问题 -> 搜索数据 -> 查询验证 -> 输出结论”的循环。

## 3. 杀手级应用

### 3.1 只读数据分析助手

这是最像第一阶段杀手应用的方向。

典型流程：

1. 用户提出业务问题
2. 外部 agent 调用 `meta search`
3. 用 `meta describe` 和 `data sample` 选表、验字段
4. 生成 SQL 并调用 `query`
5. 如果查询耗时长，则切到 `job submit / wait / result`
6. 外部 agent 解释结果并输出结论

为什么强：

- 风险低：全部是只读链路
- 需求高频：分析师、产品、运营每天都会用
- 能力闭环短：不依赖额外 AgentAPI 才能成立
- 最能体现 Agent-First CLI 的价值

### 3.2 数据事故排查助手

目标不是生成最终修复动作，而是先把“哪里出问题了”迅速定位出来。

典型问题：

- 某日报为空
- 某指标突然归零
- 某分区没有产出
- 某 SQL 任务长时间运行或失败

基础 API 足以支撑的动作：

- 查表和分区
- 看样例和 profile
- 提交诊断 SQL
- 跟踪异步任务
- 拉回结构化结果供 LLM 总结

为什么强：

- 使用场景真实
- 对团队价值非常直观
- 对 `job`、`meta`、`data` 三类命令都有直接拉动

### 3.3 SQL / 数据开发 CI 审查器

这类场景不需要 CLI 内建编排器，但非常适合 CLI。

外部 Bot 可以在 CI 中自动执行：

- `query --dry-run`
- 小样本校验 SQL
- 分区新鲜度检查
- 结果空集 / 行数异常检测
- 结构化失败归因

为什么强：

- 很容易接入现有工程流
- 结构化输出天然适合 CI 判定
- 能直接把 `maxc cli` 嵌进 PR 审核和流水线

### 3.4 可编排的数据操作工具层

即使没有 agent 模式，`maxc cli` 也可以是统一的数据操作面：

- shell 脚本调用
- Python / subprocess 调用
- Bot / Workflow 调用
- IDE 插件调用

这是生态价值，不一定是单独的“功能点”，但决定了产品上限。

## 4. 现在还不算杀手级的方向

下面这些方向重要，但不适合优先做：

- `@natural`
- CLI 内建计划 / 执行编排
- Human-in-the-Loop 审批流
- Skill Registry 市场化能力

原因不是它们不重要，而是它们建立在一个前提上：

- 底层 `query / job / meta / data` 必须已经稳定、清晰、可观测

如果基础 API 不够厚，这些高级能力只会把复杂度提前暴露出来。

## 5. 当前还差什么

从“把基础 API 做成杀手级执行底座”的角度看，当前主要缺的是下面这些能力。

### 5.1 P0：必须补

- 更短的 Agent 接入路径
  - `pip install maxc-cli` 后直接得到 `maxc` 入口
  - `auth login` / `auth whoami` 的初始化路径要稳定、可文档化
  - 对缺失身份配置的错误提示要继续打磨
- 更强的真实 backend 能力
  - 接入真实血缘 API，而不是长期停留在 unsupported 占位
  - 扩展 `auth can-i` 到更多只读操作
  - 提供更稳定的大结果导出 / 下载能力

### 5.2 P1：很快会阻塞扩展

- 语义搜索
- 缓存自动刷新
- 多凭证管理
- 更强的审计与脱敏输出

这些能力会直接影响事故排查、CI 审查和权限提示。

### 5.3 P2：可以后放

- `@natural`
- CLI 内建计划 / 执行编排
- 审批流
- Skill Registry

## 6. 产品边界

当前阶段建议明确这些边界：

- 先把 `maxc cli` 定位成只读优先
- 写操作不作为第一阶段主线
- 不把 NL2SQL 当成 CLI 的硬前置
- 不把 Skill 市场化能力当成 MVP 必选项

这有利于把产品复杂度压在一个可交付范围内。

## 7. 面向开发的直接结论

如果目标是尽快形成真实价值，接下来不应该问：

- “我们先做 agent 还是先做 skill marketplace？”

而应该问：

- “外部 agent 今天拿 `maxc cli` 能不能稳定完成一次只读分析？”
- “用户装完 `maxc-cli` 之后，能不能直接通过 `maxc auth login` 完成初始化？”
- “出事故时，Bot 能不能用 `maxc cli` 快速找出问题点？”
- “CI 能不能用 `maxc cli` 自动阻断明显危险的 SQL 变更？”

只有这三个问题的答案都越来越接近“能”，`maxc cli` 才算找到了自己的产品重心。
