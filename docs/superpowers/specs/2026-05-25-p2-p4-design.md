## 1. 目标与非目标

**目标**

- 把 SKILL 子命令从单一 `agent install-skill` 扩展为 mcs 等价的 `agent skill {install,update,uninstall,list,diff,path}` 六件套。
- 平台注册表化，新增 agent 平台仅需一行 registry。
- 加入 SKILL ↔ CLI verb 漂移自动检测。
- 修复 envelope-failure 路径 exit code 永远是 1 的不一致问题。
- 增加全局 flag hoist，让 agent 写 `maxc query "..." --json` 与 `maxc --json query "..."` 等价。
- envelope.error 补结构化 `context` 字段。
- CI 瘦身（删 GitLab CI）+ Aone 加 RUFF + coverage gate。

**非目标**

- 不做 P1（版本治理 / self-upgrade）。已确认推迟。
- 不引入 `remediation` 字段（`suggestion` 已存在，仅补 `context`）。
- 不开 MYPY（起点太高，留 P5+）。
- 不改 GitHub Actions release 触发方式（保留手动 `workflow_dispatch`）。
- 不重写 SKILL 内容主体，只做小幅结构调整。
- 不引入插件化后端协议。

## 2. P2 设计

### 2.1 命令层

```
maxc agent skill install <platform> [--dir DIR] [--invocation maxc|aliyun-maxc] [--force]
maxc agent skill update [<platform>] [--all]
maxc agent skill uninstall <platform> [--dir DIR]
maxc agent skill list [--json]
maxc agent skill diff <platform> [--unified]
maxc agent skill path [--platform X] [--source]
```

- `--invocation`：保留现有渲染机制，决定 `{{cli}}` 替换为 `maxc` 或 `aliyun maxc`。
- `--force`：覆盖目标已存在文件。
- `--all` 在 `update` 上：省略平台名时**必须显式 `--all`**才能批量；只给 `--all` 等价"遍历所有 REGISTRY 平台中检测到 `.maxc-skill-version` 的目录刷新"；裸 `update` 无平台名也无 `--all` 直接出 usage error。
- `list` 默认 plain table；`--json` 走标准 envelope。信息源：遍历 `REGISTRY` 中每个 `Platform`，按下面"effective_target"公式解析路径，存在 `.maxc-skill-version` 则视为"已安装"，输出 `{platform, install_path, installed_version_marker}`。**已知局限**：`list` 只看默认路径——若用户曾用 `agent skill install --dir CUSTOM` 装到非默认目录，`list` 不会发现该副本；envelope 输出里加 warning 提示用户传 `--dir` 自查。
- `diff` 默认输出"文件级 summary"（N 个不同 / Y 个仅本地 / Z 个仅 wheel），`--unified` 触发行级 unified diff。
- `path --source` 打印 wheel 内 `skills/` 源路径（用于脚本拼接）；不带 `--source` 时打印目标安装路径（effective_target，下面定义）。
- `update` 行为：等价 `install --force` 的子集——重新渲染并写所有 SKILL 文件（包括 `extra_files`），但**不重置** `--invocation`（沿用 `.maxc-skill-version` 里记录的 `{version}+{invocation}` 后半部分）。不做按文件级"仅变化才写"——简单可靠。

**`--dir DIR` 与平台路径的解析规则**（与 §2.2 dataclass 字段严格对应）：

```
effective_target = (--dir 给出的路径) if --dir else platform.install_root
                  / (platform.skill_subpath or "")   # 当 skill_subpath is None 时不追加任何路径段
```

- 不给 `--dir`：`effective_target = platform.install_root / (skill_subpath or "")`，与现行 `_SKILL_PLATFORMS` 取出的目录 100% 等价。
- 给 `--dir DIR`：`DIR` 直接替代 `install_root` 作为 install root，`skill_subpath` 仍按原值追加。这解锁 dev/容器化场景（如 `--dir /tmp/test`），且 claude-code 的 `extra_files`（`.claude-plugin/plugin.json`）也跟随 `--dir` 一起搬家——render_fn 收到的就是 effective install root。

### 2.2 平台注册表

新文件 `src/maxc_cli/agent_platforms.py`。**关键约束：与现有 `_SKILL_PLATFORMS` (`app.py:3374-3411`) 等价迁移，路径完全不变**——避免给已安装用户造成沉默破坏。

```python
RenderFn = Callable[[Path, str, str], None]
# (effective_install_root, cli, cli_module) -> None；callee 直接 mkdir + write_text，不返回内容
# 注意：第一参数是"effective install root"（已经叠加过 --dir override 后的最终路径），
# 不是平台的固定 install_root。
# 第二、三参数 (cli, cli_module) 当前只有 SKILL.md 模板用得上；extra_files 的 render_fn
# 若内容是固定字面量（如 plugin.json）可以忽略它们——保留在签名里是为了未来 extra_files
# 也想用 {{cli}} 占位符时无需再改协议。

@dataclass(frozen=True)
class ExtraFile:
    relative_path: str          # 相对 effective_install_root 的路径；
                                # 这是 list/diff/uninstall 用来发现此文件的"声明位置"，
                                # render_fn 必须把内容写到 `effective_install_root / relative_path`，
                                # 否则 diff 会报"文件缺失"。tests/test_agent_platforms.py 加
                                # "render_fn 写出文件正好在 relative_path"一致性测试。
    render_fn_name: str         # agent_platforms 模块内同 module callable 名；
                                # 实现签名必须匹配 RenderFn

@dataclass(frozen=True)
class Platform:
    name: str                   # canonical, e.g. "claude-code"
    install_root: Path          # 该平台的"默认"install root；不给 --dir 时使用
                                # 例：Path.home() / ".claude" / "plugins" / "maxc-cli"
    skill_subpath: str | None = None
                                # SKILL 内容相对 install_root 的子目录；
                                # None = SKILL 内容直接写到 effective_install_root
                                # （claude-code 即此情况：SKILL.md 和 .claude-plugin/ 同级）
    requires_dir: bool = True   # True = SKILL 内容是目录拷贝；False = 单文件
    extra_files: tuple[ExtraFile, ...] = ()
    aliases: tuple[str, ...] = ()
    deprecated_aliases: tuple[str, ...] = ()  # 命中时 stderr warning
    next_step_hint: str = ""    # install/update 成功后展示给用户

def _claude_plugin_root() -> Path:
    return Path.home() / ".claude" / "plugins" / "maxc-cli"

def _codex_root() -> Path:
    return Path(os.environ.get("CODEX_HOME", str(Path.home() / ".codex"))) / "skills" / "maxcompute-cli-guidance"

REGISTRY: tuple[Platform, ...] = (
    Platform(
        "claude-code",
        install_root=_claude_plugin_root(),
        skill_subpath=None,
        extra_files=(ExtraFile(".claude-plugin/plugin.json", "render_claude_plugin"),),
        next_step_hint="Run /reload-plugins in Claude Code to activate",
    ),
    Platform("cursor",    install_root=Path.home() / ".cursor"    / "skills" / "maxcompute-cli-guidance",
             next_step_hint="Restart Cursor to activate"),
    Platform("windsurf",  install_root=Path.home() / ".windsurf"  / "skills" / "maxcompute-cli-guidance",
             next_step_hint="Restart Windsurf to activate"),
    Platform("codex",     install_root=_codex_root(),
             next_step_hint="Restart Codex to activate"),
    Platform("qwen",      install_root=Path.home() / ".qwen"      / "skills" / "maxcompute-cli-guidance",
             next_step_hint="Restart Qwen to activate"),
    Platform("qoder",     install_root=Path.home() / ".qoder"     / "skills" / "maxcompute-cli-guidance",
             next_step_hint="Restart Qoder to activate"),
    Platform("qoderwork", install_root=Path.home() / ".qoderwork" / "skills" / "maxcompute-cli-guidance",
             next_step_hint="Restart QoderWork to activate"),
)

def render_claude_plugin(install_dir: Path, cli: str, cli_module: str) -> None:
    """Write .claude-plugin/plugin.json under install_dir.

    The output path MUST equal install_dir / Platform.extra_files[i].relative_path
    (i.e. ".claude-plugin/plugin.json") — diff/list discover files via relative_path,
    so render_fn must agree. See test_agent_platforms.py::test_render_fn_writes_declared_path.

    To avoid spurious diff on first upgrade from existing installs, keep the exact byte
    output of app.py:3493-3499 (a single-line author object). Implementation should use
    a literal string template, not json.dumps with indent (which would re-flow the JSON
    and cause `agent skill diff` to report a phantom delta on day-1).
    """
    meta_dir = install_dir / ".claude-plugin"
    meta_dir.mkdir(parents=True, exist_ok=True)
    (meta_dir / "plugin.json").write_text(
        # Byte-equivalent to the literal in app.py:3493-3499 (do not reformat).
        '{\n  "name": "maxc-cli",\n'
        '  "description": "MaxCompute/ODPS CLI — query tables, view schema, '
        'search metadata, execute SQL, check partitions, sample data, '
        'track jobs. Install via: pip install maxc-cli",\n'
        '  "author": { "name": "maxc-cli contributors" }\n}\n',
        encoding="utf-8",
    )

def resolve(name_or_alias: str) -> Platform: ...
def all_platforms() -> tuple[Platform, ...]: ...
```

## 3. P4 设计

### 3.1 全局 flag hoist

新函数 `cli.py:_hoist_global_flags(argv: list[str]) -> list[str]`，在 `run()` argparse 解析前调用。

被 hoist 的 flag 列表（与主 parser 的 global flag 集合保持同步）：

```
--format, -f, --json, --quiet, -q, --debug, --config, -v, --version, --help, -h
```

规则：

- 同时支持 `--flag value`、`--flag=value`、单字 flag 三种形式。
- 遇 `--`（POSIX 终止符）立即停止 hoist，之后内容原样保留。
- 子命令的位置参数（如 query 后的 SQL 字符串）整体不动，flag 仅从位置参数后被前移；位置参数内部字面量（如 SQL 含 `--json`）不被识别为 flag——因为它紧跟位置参数之后，argv 处理时已经被当成参数值。

**Flag arity 表**（决定 `--flag value` 空格式何时取下一个 argv 作为值）：

| flag | arity | 取值类型 |
|---|---|---|
| `--format` / `-f` | 1 | `json` \| `table` \| `csv` \| `ndjson` \| `markdown` \| `brief` |
| `--config` | 1 | path |
| `--json` | 0 | bool |
| `--quiet` / `-q` | 0 | bool |
| `--debug` | 0 | bool |
| `-v` / `--version` | 0 | bool |
| `--help` / `-h` | 0 | bool |

实现：hoist 函数内置上述 arity 表常量；对 arity=1 的 flag 看 `--flag=val` 时直接拆，看裸 `--flag` 时把下一个 argv 也一起搬。arity=0 的 flag 不消耗后续 argv。

新测试 `tests/test_flag_hoist.py` 覆盖：

| 输入 | 期望 |
|---|---|
| `maxc query "select 1" --json` | 等价 `maxc --json query "select 1"` |
| `maxc --json query "select 1"` | 不变 |
| `maxc query "select '--json'"` | `--json` 在引号内，不被 hoist |
| `maxc -- query "select 1" --json` | `--` 后不动 |
| `maxc --config=/tmp/a.yaml query x` | `--config=/tmp/a.yaml` 已在主位置 |
| `maxc query x --config=/tmp/a.yaml` | hoist 后等价上一条 |

### 3.2 exit code 路径统一

当前状态：

- `exceptions.py:69-143` 每个异常类已有 `exit_code` ClassVar（PermissionDeniedError=2, QuotaExceededError=3, SqlError=4, CostLimitExceededError=5…）。
- `cli.py:872` 异常抛出路径正确返回 `exc.exit_code`。
- `cli.py:1610` envelope-failure 路径永远写 `_envelope_exit_code = 1`。

修复方案——**在 `ErrorPayload` 上新增 `exit_code: int = 1` 字段**（替代靠 `error.code` 字符串反查表）：

1. `exceptions.py:ErrorPayload` 加 `exit_code: int = 1`，`to_dict()` 序列化时**不输出**该字段（保持 envelope schema 向后兼容；它只为 CLI 内部决定 exit 用）。
2. `MaxCError.to_payload()` 把 `self.exit_code` 写进 payload。
3. `cli.py:1610` 改为：从 envelope.error 取 `exit_code`（通过私有 attribute 或在 `_emit_envelope` 内对 payload 对象本身取值，不读 dict），失败兜底 1。
4. 不引入字符串 → exit_code 的 lookup 表（重复维护成本）。

矩阵：

| 来源 | 当前 | 改为 |
|---|---|---|
| 抛 `MaxCException` 顶层捕获 | `exc.exit_code` | 不变 |
| 命令显式返回 failure envelope | 1 | envelope.error.exit_code（异常类 default 1） |
| success / pending envelope | 0 | 不变 |
| 顶层意外异常 | 1 | 不变 |

新测试 `tests/test_exit_codes.py`：mock 每个异常类 → 走 envelope 路径 → 断言 exit code。

### 3.3 envelope.error.context 字段（扩展现有用法）

**现状校对**：`exceptions.py:7-30` 已定义 `ErrorPayload.context: dict | None`，序列化在 `to_dict()` 第 28-29 行。仅 `CsvParseError`（`exceptions.py:142-166`）通过自定义 `to_payload()` 用了它，其他 14 个异常子类全都不传 `context`。

因此本节不是"新增字段"，而是**让基类用起来 + 把现有散落的 schema-context 注入归位**：

1. `exceptions.py:MaxCError.__init__` 增加可选 `context: dict[str, Any] | None = None` 参数；非 None 时存 `self.context` 并在默认 `to_payload()` 中写入 `ErrorPayload.context`（替代/简化 CsvParseError 的覆写）。
2. `cli.py:552-635` 当前在 envelope failure 时把 schema 信息塞到 `metadata.schema_context` / `agent_hints` 散落位置。改为：构造异常时直接 `raise TableNotFoundError(msg, context={"table": ..., "project": ..., "schema": ...})`。原 metadata 注入逻辑改为 fallback——异常已带 context 时不再做二次注入。
3. envelope schema `version` 仍保持 `2.0`。旧消费者忽略未知字段，新消费者优先读 `error.context`。

新测试覆盖：
- `TableNotFoundError("not found", context={"table":"x", "project":"p"})` → `envelope.error.context == {"table":"x", "project":"p"}`
- 现有 `CsvParseError` 行为不破坏（其 context 仍正确序列化）
- `cli.py:552-635` 路径：构造一个会触发 schema-context 注入的失败场景，断言信息出现在 `error.context` 而不是 `metadata`

## 4. 数据流（关键变更点）

```
argv ──► _hoist_global_flags ──► argparse ──► handler ──► MaxCApp.method
                                                                │
                                                                ├── 抛 MaxCException ─► cli.run 顶层 catch ─► exit(exc.exit_code)
                                                                │
                                                                └── 返回 Envelope ─► _emit_envelope
                                                                                            │
                                                            success/pending ─────► exit 0
                                                                                            │
                                                            failure ─► error.exit_code（来自原异常类，default 1） ─► exit(N)

agent skill <verb> ──► agent_platforms.resolve(name) ──► Platform spec ──► MaxCApp.skill_{install,update,...}
                                                                                            │
                                                                                            └─► 写 .maxc-skill-version + 渲染 extra_files
```
